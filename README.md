# SDD-mini

## TL;DR

**Что это.** Пять слеш-команд для Claude Code, которые ведут от идеи до работающего MVP **веб-приложения**: интервью → `PROJECT.md` → сборка по фазам, один коммит и тесты на каждую фазу.

**Что строим.** Только веб-приложения на одном стеке: **Django 5 + htmx + SQLite + Pico.css в Docker**. Стек не обсуждается и не выбирается — он один на всё.

**Чего НЕ строим.** Мобильные приложения, десктоп-приложения, CLI-утилиты, нативные игры. Если Ваша идея из этой группы — попробуйте сначала собрать веб-версию через SDD, а позже обернуть её в PWA / Tauri / нативный клиент.

**Команды:**

- `/sdd-idea` — идея → `PROJECT.md` со спецификацией и фазами.
- `/sdd-impl` — построить следующую фазу (коммит + тесты в конце).
- `/sdd-undo` — откатить последнюю фазу через `git revert` (без перезаписи истории).
- `/sdd-feature` — добавить новую фичу в существующий план.
- `/sdd-change` — изменить спецификацию, будущую фазу или уже построенное поведение.

**Установка.** `git clone` этого репозитория, `./install.sh`, затем `/sdd-idea` в пустой папке проекта.

---

**SDD (Spec-Driven Development) для vibe coders** — пять скиллов для Claude Code, которые превращают идею в веб-MVP: одно длинное интервью, затем серия итераций реализации — один коммит на фазу, тесты на каждое изменение.

Стек один и не обсуждается: Django + htmx в Docker. Фаза 1 поднимается автоматически — SDD создаёт файлы, поднимает контейнер, проверяет, что главная страница и админка открываются. Дальше каждая фаза добавляет одну вертикальную фичу: модель + вьюха + шаблон + тесты + коммит.

## Кому это подходит

Проект подходит Вам, если:
- У Вас есть идея **веб-приложения**, которую нужно быстро поднять и при необходимости расширять.
- Вы умеете пользоваться терминалом на уровне «открыть папку, выполнить команду».
- Вы готовы не спорить со стеком — он один (Django + htmx), это сэкономит время.
- Вы принимаете, что Claude пишет бóльшую часть кода, а Вы ведёте проект, проверяете результат и даёте обратную связь.

Примеры идей, которые SDD закрывает:
- Трекеры: книги, фильмы, тренировки, привычки, расходы.
- Менеджеры: рецепты, бюджет, закладки, проекты, задачи.
- Внутренние инструменты: чек-листы, простые CRM, базы знаний для команды.
- Простые блоги, лендинги с админкой, формы с результатами.

Что **не подойдёт**: 3D-игры, мобильные приложения с push-уведомлениями и камерой как ядром продукта, kernel-модули, торговые боты с жёсткими требованиями к латенси, что-то требующее GPU/нативной графики. Для большинства «мобильных» идей `/sdd-idea` предложит начать с веб-версии — её можно открыть с телефона, а позже обернуть в PWA или Capacitor.

## Что Вы получаете

После прогона всех фаз у Вас будет:

- **Работающий веб-MVP** в Docker — `docker compose up`, и приложение открывается в браузере.
- **Стек продакшен-уровня** — Django 5 + htmx + SQLite + Pico.css. Тот же стек используется в боевых проектах, поэтому MVP можно расширять без переписывания.
- **Админка из коробки** — Django даёт её бесплатно, можно править данные через `http://localhost:5000/admin/`.
- **100% покрытие тестами** на файлах, которые изменялись в ходе разработки. Это предотвращает ситуацию, когда следующая фаза незаметно ломает предыдущую. Тесты запускаются командой из `PROJECT.md` (django test). Фаза не коммитится, пока покрытие не зелёное.
- **История git** — один коммит на фазу (`phase 1: setup`, `phase 2: ...`). Любую фазу можно откатить командой `/sdd-undo`.
- **`PROJECT.md`** в корне — единый файл со спецификацией, инструкцией по сборке Фазы 1 и планом по фазам с галочками. При возвращении к проекту через месяц Claude по нему восстанавливает контекст.
- **`CLAUDE.md`** — короткий указатель на `PROJECT.md`, чтобы Claude в следующих сессиях не тащил в проект чужой стек.
- **Деплой** — последняя фаза плана (Render.com или Fly.io).

## Стек

SDD ставит и собирает один стек, без вариантов:

| Слой | Технология | Зачем |
|---|---|---|
| Бэкенд | **Django 5** | Админка, ORM, формы, миграции, аутентификация — всё из коробки. |
| Интерактивность | **htmx** | Drag-and-drop, инлайн-редактирование, живые обновления — без React и JS-сборки. |
| База | **SQLite** | Один файл, нулевая настройка, годится для MVP и небольших команд. |
| Стили | **Pico.css** | Classless-CSS — выглядит прилично без вёрстки. |
| Контейнер | **Docker** | Никакого Python на хосте, только `docker compose up`. |
| Тесты | **django.test + coverage** | Встроенный тест-раннер, 100% покрытие как ворота фазы. |

Этот стек подходит для всего из списка «Кому это подходит» выше. Если идея просит чего-то ещё (Postgres, Redis, очереди, мульти-регион, мобильный клиент) — это идёт в раздел «Будущее расширение» в `PROJECT.md` и в более поздние фазы, не в Фазу 1.

## Что нужно установить заранее

На macOS первый запуск `/sdd-impl` предложит поставить недостающее автоматически: если у Вас есть Homebrew, он поставит git и Docker Desktop без Вашего участия; если Homebrew нет — откроет системный диалог установки для git и даст ссылку на Docker Desktop. На Linux пока только вручную по инструкциям ниже.

### 1. Docker Desktop

Docker запускает приложение в изолированном контейнере. SDD использует его, чтобы Django работал одинаково на любой машине — на хосте Python ставить не нужно.

Скачайте с [docker.com/products/docker-desktop](https://www.docker.com/products/docker-desktop/) и установите. Запустите приложение и дождитесь, пока индикатор статуса перейдёт в состояние «Running».

На Linux вместо Docker Desktop подойдёт Docker Engine + docker-compose-plugin:
```bash
sudo apt install docker.io docker-compose-plugin   # Debian/Ubuntu
sudo dnf install docker docker-compose-plugin      # Fedora
```

### 2. Git

Git — это система контроля версий, которая хранит историю всех изменений в коде. SDD делает один коммит на каждую фазу; благодаря этому любую неудачную фазу можно откатить командой `/sdd-undo` и вернуться к работающему состоянию.

Как правило, уже установлен. Проверка:
```bash
git --version
```

Если нет:
- macOS: `xcode-select --install`
- Linux: `sudo apt install git` или `sudo dnf install git`

### 3. Claude Code

Claude Code — это терминальный клиент для работы с AI-моделью Claude: Вы общаетесь с ней текстом, а она читает и пишет файлы в Вашей папке. SDD-скиллы встраиваются в Claude Code и запускаются прямо из сессии.

Установите его по инструкции с [claude.com/claude-code](https://claude.com/claude-code).

### 4. Плагин `frontend-design`

Плагин добавляет Claude набор правил и примеров хорошего UI — без него сгенерированный интерфейс выглядит как шаблонный AI-вывод. SDD требует его наличия: `/sdd-impl` не начнёт сборку, пока плагин не установлен.

В Claude Code выполните:
```
/plugin marketplace add anthropics/claude-code
/plugin install frontend-design@claude-plugins-official
```

Затем перезапустите Claude Code.

### 5. `agent-browser` (для проверок UI)

`/sdd-impl` после каждой фазы прогоняет визуальную проверку: открывает приложение в реальном браузере, кликает по кнопкам как пользователь, делает скриншоты и сверяет результат с чекпоинтом фазы из `PROJECT.md`. Это ловит то, что `curl` не видит — сломанный тёмный фон, не подгрузившийся CSS, htmx-фрагмент не туда подменился, кнопка стала невидимой.

Без `agent-browser` SDD сделает только базовую HTTP-проверку (`curl`) и сообщит об этом — фаза всё равно закоммитится. Но визуальные баги придётся ловить руками, поэтому ставить рекомендуется.

Установка (одна из):

```bash
npm install -g agent-browser     # любая ОС, нужен Node.js + npm
brew install agent-browser       # только macOS
```

Затем при первом запуске нужно один раз скачать встроенный Chrome:

```bash
agent-browser install
```

Подробности и альтернативные способы установки: [agent-browser.dev](https://agent-browser.dev/).

### Поддержка ОС

**macOS и Linux.** Windows (нативный или WSL) не поддерживается — из-за множества мелких проблем с путями и томами, которые будут отвлекать от работы. Если Вы на Windows, используйте виртуальную машину с Ubuntu.

## Установка SDD

В терминале:

```bash
git clone <url-этого-репозитория>
cd SDD-mini
./install.sh
```

Установщик скопирует скиллы на место, проверит окружение и, если чего-то не хватает (Docker, git, плагина), сообщит что именно и как исправить. Старые версии скиллов сохранятся в резервной копии.

После установки **перезапустите Claude Code**, чтобы он подхватил новые команды.

## Первый проект за 30 минут

Допустим, Вы хотите сделать трекер книг: вносить прочитанное, ставить оценки, писать короткие заметки.

### Шаг 1. Пустая папка

```bash
mkdir ~/projects/book-tracker
cd ~/projects/book-tracker
```

### Шаг 2. `/sdd-idea` — спроектировать

Откройте Claude Code в этой папке и напишите:

> `/sdd-idea` Хочу приложение для учёта прочитанных книг: оценка, короткая заметка, статистика по месяцам.

Claude начнёт **интервью на 8–12 вопросов**: кто пользователи, как выглядит типичная сессия, какие данные хранятся, нужен ли логин, на что похоже визуально, какие три фичи самые важные. Отвечайте по существу — Вам виднее, что нужно.

Если идея — мобильное / десктоп / CLI приложение, Claude один раз спросит, подойдёт ли веб-версия (открыть с телефона, обернуть в PWA позже). Если согласны — продолжаем. Если строго нужна нативная сборка — Claude скажет, что SDD это не закроет, и выйдет.

После интервью Claude предложит MVP — список того, что имеет смысл собрать первым. Если состав устраивает — подтвердите. Если нет — укажите, что убрать или добавить.

После одобрения в папке появится **`PROJECT.md`** — спецификация, стек с инструкцией по сборке Фазы 1 и список из ~5 фаз.

Claude предложит три варианта: продолжить интервью, поменять план или начать реализацию. Если нужен ещё раунд вопросов — напишите «продолжим интервью». Если хотите поменять фазы местами или вырезать что-то — «поменять план». Когда всё устраивает — «начинаем реализацию».

### Шаг 3. `/sdd-impl` — Фаза 1

В терминале (или в той же сессии Claude Code):

> `/sdd-impl`

Claude развернёт проект по плану из `PROJECT.md`: создаст все нужные файлы, поднимет контейнер, поднимет Django, напишет первые тесты и закоммитит всё как `phase 1`. В конце Вы получите **ссылку на работающее приложение** и сможете открыть его в браузере по `http://localhost:5000`.

### Шаг 4. `/sdd-impl` — следующие фазы

Каждая фаза добавляет **одну вертикальную фичу** (не «все модели сначала, потом все вьюхи», а «фича целиком: модель + вьюха + шаблон + тесты»).

> `/sdd-impl`

Например, Фаза 2 может быть «добавление книги»: модель `Book`, форма, список на главной, тесты. После фазы — новый коммит, приложение работает, и Вы можете вручную добавить книгу в браузере.

Повторяйте `/sdd-impl`, пока все фазы не станут `[x]`. Последняя фаза — всегда деплой на Render.com или Fly.io.

### Шаг 5. Что-то пошло не так

Фаза получилась неудачной — интерфейс выглядит плохо, логика работает некорректно или тесты покрывают код формально:

> `/sdd-undo`

Claude откатит последнюю фазу через `git revert` (история не перепишется — это безопасно). Задачи фазы в `PROJECT.md` снова станут `[ ]`. Далее — либо сразу запустите `/sdd-impl`, чтобы попробовать заново, либо отредактируйте `PROJECT.md` вручную (например, перепишите чекпоинт фазы) и затем запустите `/sdd-impl`.

### Шаг 6. Добавить что-то новое

Проект уже работает, и Вам нужна новая фича — например, делиться книгами через публичную ссылку:

> `/sdd-feature` Хочу, чтобы можно было делиться любой книгой через публичную ссылку, без логина.

Claude задаст 2–3 вопроса (где в UI, как выглядит публичная страница, нужен ли срок жизни ссылки) и допишет новые фазы в `PROJECT.md`. Предыдущий план сохраняется в `PROJECT.v<N>.md`. Далее — `/sdd-impl`, как обычно.

### Шаг 7. Поменять то, что уже запланировано или построено

Передумали про какую-то часть — описание в спеке, задачи в будущей фазе, или даже поведение того, что уже работает:

> `/sdd-change` Хочу, чтобы логин был по email, а не по телефону.

Claude посмотрит, построено ли это уже в коде, и выберет правильный вариант сам:
- если это только в спеке или в ещё не построенной фазе — перепишет соответствующий кусок `PROJECT.md` на месте;
- если код уже есть — допишет **новую фазу миграции** в конец плана. Завершённые фазы с галочками `[x]` не трогаем — они точно соответствуют тому, что лежит в git.

Предыдущий план сохраняется в `PROJECT.v<N>.md`. Если появилась новая фаза или переписана будущая — запустите `/sdd-impl`, чтобы её собрать.

## Пять команд — справочник

| Команда | Что делает | Когда запускать |
|---|---|---|
| `/sdd-idea` | Интервью → `PROJECT.md`. Никакого кода. | В пустой папке, когда есть идея. |
| `/sdd-impl` | Строит следующую невыполненную фазу. Коммит в конце. | Когда есть `PROJECT.md`. Запускайте по одной фазе за раз. |
| `/sdd-undo` | Откатывает последнюю фазу через `git revert`. | Когда фаза получилась неудачной. |
| `/sdd-feature` | Интервью → новые фазы в `PROJECT.md`. Кода не пишет. | Когда проект уже есть и нужна новая фича. |
| `/sdd-change` | Меняет план или спеку в `PROJECT.md`. Если изменение касается уже построенного кода — появится новая фаза миграции. Кода не пишет. | Когда передумали про что-то из спеки или будущей фазы, или когда нужно изменить поведение того, что уже работает. |

Триггеры `/sdd-idea`, `/sdd-impl`, `/sdd-undo`, `/sdd-feature`, `/sdd-change` нужно писать явно — по одной фразе «поменяй план» или «откати» скилл не запустится, чтобы не делать ничего случайно.

## Как устроен проект

После Фазы 1 в Вашей папке появится стандартный Django-проект в Docker. Все необходимые команды (запуск, тесты, миграции, логи) — в `PROJECT.md` и коротком `CLAUDE.md` в корне Вашего проекта.

## На чём держится весь подход

- **`PROJECT.md` — источник истины.** Спецификация, стек, план, галочки по задачам — в одном файле. При возвращении к проекту через месяц Claude откроет его и восстановит контекст.
- **Один коммит на фазу.** История git служит безопасным механизмом отката. `/sdd-undo` делает revert-коммит (не переписывает историю), его можно запускать подряд.
- **100% покрытия на изменённых файлах.** Без тестов следующая фаза незаметно ломает предыдущую; с тестами — не может. Фаза не коммитится, пока тесты не зелёные.

## FAQ и типичные проблемы

### Claude не видит команды `/sdd-*`

- Перезапустите Claude Code после установки.
- Проверьте, что папки появились: `ls ~/.claude/skills/sdd-*`. Если нет — запустите `./install.sh` из папки репозитория.

### Нужен плагин `frontend-design`

В Claude Code:
```
/plugin marketplace add anthropics/claude-code
/plugin install frontend-design@claude-plugins-official
```
Перезапустите Claude Code и попробуйте снова.

### `/sdd-impl` пишет «agent-browser недоступен — делаю расширенный curl-чек»

Это не блокер: фаза закоммитится по итогам `curl`-проверки. Но визуальные косяки (сломанный тёмный фон, съехавшая вёрстка, htmx-фрагмент не туда подменился) `curl` не ловит. Поставьте `agent-browser` и в следующей фазе проверка будет полноценной:

```bash
npm install -g agent-browser     # или: brew install agent-browser
agent-browser install
```

См. [agent-browser.dev](https://agent-browser.dev/).

### Docker не запущен

- macOS: `open -a Docker`, подождите 10–15 секунд.
- Linux: `sudo systemctl start docker`.
- Если Docker установлен впервые — возможно, потребуется перезагрузка.

### `/sdd-impl` сообщает «сначала нужен `/sdd-idea`»

В папке нет `PROJECT.md`. Запустите `/sdd-idea` и спроектируйте проект.

### Фаза получилась неудачной, но я уже закоммитил

`/sdd-undo`. Это безопасно: используется `git revert`, а не `reset`. Команду можно запускать несколько раз подряд, чтобы откатиться на несколько фаз назад.

### `PROJECT.md` выглядит странно, хочу отредактировать его вручную

Это допустимо. Файл — обычный Markdown. Не меняйте задачи в **уже завершённых фазах** (`- [x]`) — они соответствуют реальному коду в git. В будущих фазах можно менять что угодно. Перед большой правкой сделайте резервную копию: `cp PROJECT.md PROJECT.v-manual.md`.

### Хочу мобильное приложение / CLI / десктоп

SDD только веб. Если идея в основе — данные + формы + список + отчёты, она спокойно живёт как веб-приложение, открывается с телефона, и позже её можно обернуть в PWA или Capacitor. Запустите `/sdd-idea`, опишите идею — скилл сам предложит реалистичный путь. Если нативная сборка обязательна (например, нужны фоновые геолокационные пуши) — SDD не подойдёт, ищите другой инструмент.

## Обновление и удаление

### Обновить SDD

```bash
cd SDD-mini
git pull
./install.sh
```

`install.sh` просто перезаписывает скиллы в `~/.claude/skills/` свежими версиями из репозитория. Если Вы правили что-то руками в `~/.claude/skills/sdd-*`, эти правки пропадут — храните свои изменения в форке этого репозитория, а не в установленной копии.

### Удалить SDD

```bash
rm -rf ~/.claude/skills/sdd-idea ~/.claude/skills/sdd-impl ~/.claude/skills/sdd-undo ~/.claude/skills/sdd-feature ~/.claude/skills/sdd-change
# (на случай, если осталась старая копия скрипта-доктора)
rm -f ~/.claude/scripts/sdd-doctor.sh
```

Уже построенные проекты не пострадают — они самодостаточны: весь план и инструкции по сборке находятся в `PROJECT.md`.

## Структура репозитория

```
SDD-mini/
├── README.md                  ← этот файл
├── install.sh                 ← копирует скиллы в ~/.claude/
└── skills/
    ├── sdd-idea/
    │   ├── SKILL.md
    │   └── references/
    │       └── django-htmx.md ← единственный стек, инлайнится в PROJECT.md
    ├── sdd-impl/
    │   ├── SKILL.md
    │   └── scripts/
    │       └── sdd-doctor.sh  ← проверка окружения
    ├── sdd-undo/SKILL.md
    ├── sdd-feature/SKILL.md
    └── sdd-change/SKILL.md
```

Если нужно изменить поведение скилла под себя — откройте соответствующий `SKILL.md`, внесите правки и запустите `./install.sh` заново. Скиллы — это Markdown-файлы с инструкциями для Claude.

## Коротко

1. Установите Docker, git, Claude Code, плагин `frontend-design`, `agent-browser` (для визуальных проверок).
2. `git clone` этого репозитория, `./install.sh`.
3. В пустой папке запустите `/sdd-idea` и ответьте на вопросы. SDD строит веб-приложение на Django + htmx.
4. Запускайте `/sdd-impl` по одной фазе, пока MVP не готов.
5. `/sdd-undo` — если что-то пошло не так. `/sdd-feature` — когда нужна новая фича. `/sdd-change` — когда нужно поменять описание, будущую фазу или уже построенное поведение.
